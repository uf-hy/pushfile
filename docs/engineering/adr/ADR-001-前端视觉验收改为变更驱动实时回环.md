# ADR-001：前端视觉验收改为变更驱动实时回环

- 状态：已采纳
- 日期：2026-03-23

## 背景

之前的前端视觉验收主链路过度依赖固定 plan / 固定 state：

- 先预设页面状态
- 再由脚本执行有限动作
- 然后把少量截图交给视觉模型
- 最后根据结构化 `pass` 推断页面整体通过

这套方法在本仓库中已真实失效，典型症状包括：

- `manage-grid-desktop-loaded.png` 已经生成，但没有进入正式审查
- `manage-album-desktop-current.png` 存在明显问题，但也没有进入正式审查
- `default` 状态通过，被误外推成页面整体通过
- `evidence/` 被误当成长期知识来源，实际只是运行产物

## 决策

从现在开始，前端视觉验收主链路改为：

1. **变更驱动**：先看本次改动影响了哪些页面/模块
2. **实时交互**：由 agent 像人类一样直接操作页面，不依赖固定 plan 回放
3. **用户路径优先**：围绕真实用户路径生成截图，而不是围绕脚本预设状态生成截图
4. **关键状态截图**：desktop / mobile + 默认态 / 当前态 / 交互态都按页面特性覆盖
5. **`read` 为主审图工具**：关键截图默认使用 `read` 直接审图
6. **`look_at` 为辅助工具**：只做快速第二视角，不再作为放行依据
7. **证据与审查强绑定**：任何关键截图如果没有被明确审过，视为未审
8. **visual-loop 降级**：保留为补充产图 / 留档工具，不再作为主放行链路
9. **用户任务优先**：验收必须先看主用户任务和首屏反馈，再看局部组件问题
10. **堆叠问题升级**：同一主任务路径上 3 个以上轻中度问题，按系统级失败处理

## 影响

### 正向影响

- 更接近真实用户体验
- 更容易发现 current / loaded / interaction 态问题
- 更符合 agentic / context-engineering 的 just-in-time 检索和执行方式
- 减少“脚本没看到，所以问题不存在”的假安全感

### 代价

- 单次验收比固定脚本更费人工判断
- 必须补足可检索的长期文档，否则新会话仍会遗忘
- 需要对每个复杂页面维护状态矩阵和已知坑位

## 落地约束

- `screenshot exists != reviewed`
- `default pass != page pass`
- `.sisyphus/evidence/` 只存证据，不存长期真相
- 长期真相必须进入 `docs/engineering/`
