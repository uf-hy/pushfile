# 视觉验收失效模式

## 目的

记录已经真实发生过的漏审与误判模式，避免未来重复踩坑。

## 失效模式 1：截图存在，但没有进入正式审查

### 表现

- 图已经生成并归档
- 但没有被明确审图
- 最后却因为“图在 evidence 里”而产生已审错觉

### 典型案例

- `manage-grid-desktop-loaded.png`
- `manage-album-desktop-current.png`

### 防线

- 每张关键图必须能回答“谁审了、怎么审的、结论是什么”
- 没有明确审图记录就视为未审

## 失效模式 2：`default pass` 被误外推成页面整体 pass

### 表现

- 只审了 `default`
- loaded / current / interaction 态没审
- 但结论被写成“该页面没问题”

### 防线

- 页面必须按状态矩阵验收
- 必须在汇报时明确“审了哪些状态”

## 失效模式 3：结构化 report 制造假安全感

### 表现

- `report.json` 显示 `pass`
- 人误以为页面整体通过
- 实际它只说明 plan 中那几个状态没被模型指出明确问题

### 防线

- 不以单个 report 的 `pass` 作为放行依据
- 只把 report 当附加证据，不当最终裁决

## 失效模式 4：固定 plan / 固定脚本覆盖不了真实改动路径

### 表现

- 脚本只会走预设动作
- 本次改动真正危险的状态并不在脚本里
- agent 最终审到的是“脚本理解的页面”，不是“真实被改动的页面”

### 防线

- 主链路改为变更驱动实时交互
- 先看改了什么，再决定看什么

## 失效模式 5：整页图掩盖局部灾难

### 表现

- 整页看似“还行”
- 某个边栏、参数区、工具条已经局部崩坏
- 模型或人工快速扫视时容易漏掉

### 防线

- 一旦整页中已露头，就必须补局部图
- 局部图同样要进入正式审查

## 失效模式 6：把 `look_at` 当最终裁判

### 表现

- `look_at` 给出一个大致摘要
- 人误把摘要当精确审图结论

### 防线

- `look_at` 仅做辅助
- 关键截图一律用 `read` 主审

## 失效模式 7：把证据目录当长期知识库

### 表现

- 截图、报告、测试输出很多
- 但没有提炼成可复用结论
- 新会话仍然重复犯同样错误

### 防线

- 结论进入 `docs/engineering/`
- 当前任务发现先写 `.sisyphus/notepads/`
- 稳定后提升为 ADR / playbook / eval / pitfalls

## 失效模式 8：只修局部 bug，没有审主任务路径

### 表现

- 已经修掉某个具体小 bug
- 但页面的主用户任务仍然不顺
- 最终页面“技术上更正确”，体验上仍然错误

### 典型案例

- 手机端 `/manage/grid` 上传后，虽然局部竖排问题被修复，但首屏依旧没有把成品预览放在最优先位置

### 防线

- 验收顺序必须先看任务层，再看信息架构层，最后才看局部实现层
- 放行前必须回答“用户完成关键动作后，第一屏是不是先看到了最该看到的结果”

## 失效模式 9：把多个小问题拆散看，错过系统级失败

### 表现

- 每个问题单独看都像轻微瑕疵
- 但它们共同压在同一条主任务路径上
- 用户整体体验已经明显变差，审查却仍把它们当成零散小问题

### 防线

- 引入“小问题堆叠升级规则”
- 同一路径上的 3 个以上轻中度问题，按系统级失败处理
