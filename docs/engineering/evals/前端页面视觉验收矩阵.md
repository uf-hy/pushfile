# 前端页面视觉验收矩阵

## 通用规则

### 关键定义

- **已截图**：图已经生成
- **已审图**：该图已由 `read` 明确审过
- **已通过**：关键状态全部已审，且没有阻断问题

> 规则：**已截图 ≠ 已审图，已审图 ≠ 页面整体已通过。**

### 通用最低覆盖

对任意被本次改动影响的前端页面，至少覆盖：

- desktop 关键状态
- mobile 关键状态
- 至少一个当前态 / interaction 态

### 用户视角硬门槛

每个受影响页面在汇报“通过”前，必须回答：

1. 目标用户是谁
2. 主任务是什么
3. 关键动作后的第一屏应该看到什么
4. 现在第一屏是否真的先展示了这个结果

如果主结果没有出现在最该被看到的位置，直接视为**未通过**。

### 小问题堆叠升级规则

如果 3 个以上轻中度问题共同压在同一条主任务路径上，应升级为一个**系统级失败**。

典型组合：

- 结果不在首屏
- 控件抢占主视觉焦点
- 文案换行 / 挤压
- 入口重复或主次混乱
- 局部对齐/留白异常

### 局部截图触发条件

出现以下任一情况时，必须补局部截图：

- 有明显局部挤压，但整页图不够清楚
- 顶部工具条/路径栏/按钮组密度过高
- 左侧控制区或边栏疑似窄宽度失衡
- 文字疑似逐字换行、截断、对比度过低

## 页面矩阵

### `/manage/grid`

当改动涉及 `grid.*`、九宫格逻辑、参数控件、预览区、导出区时，至少覆盖：

1. desktop 默认态
2. mobile 默认态
3. desktop 上传图片后
4. mobile 上传图片后
5. desktop 调整参数后
6. mobile 调整参数后
7. 如参数区曾出过问题：desktop 左侧参数区局部图

重点检查：

- 上传图片后，首屏是否优先展示成品预览而不是参数区
- 手机端调参是否采用不会强迫用户大幅上下往返的模式（优先考虑预览主舞台 + 参数底部抽屉/编辑器）
- 左右工作区比例
- 参数区可读性
- 中文标签是否逐字换行
- 预览区与九宫小图区层级
- 默认值 / 调参后的反馈是否清晰

### `/manage/album`

当改动涉及 `manager/*`、搜索、下载模式、列表/网格、路径栏、Finder 化壳层时，至少覆盖：

1. desktop 当前数据态
2. mobile 当前数据态
3. desktop 搜索后
4. desktop 选择文件/文件夹后（如本次涉及选择态）
5. desktop 切换下载模式后（如本次涉及下载语义）
6. 如顶部操作区复杂：desktop 顶部工具条局部图

重点检查：

- 当前态首屏是否优先服务“浏览 / 搜索 / 选择 / 下载”主任务
- 顶部操作是否重复或主次混乱
- 当前路径与工具条层级是否顺
- 空态 / 子文件夹 / 列表内容关系是否清楚
- 左侧列表长标题是否过度截断
- 当前态是否存在大面积无意义留白

### `/manage/dashboard`

当改动涉及 `dashboard.*`、统计卡片、图表、列表、标题 fallback 时，至少覆盖：

1. desktop 默认态
2. mobile 默认态
3. 如本次改到数据呈现：一张有真实数据的状态图

重点检查：

- 标题是否泄露 technical/hash-like 字段
- 图表与文案语义是否一致
- mobile 标签与图例是否可读

### 首页 / 公共页

当改动 landing、公开页、登录页时，至少覆盖：

1. desktop 首屏
2. mobile 首屏
3. 如改到交互入口，再补一次交互后状态

重点检查：

- 首屏留白是否异常
- CTA 是否在第一视线内
- 文字层级是否稳定

## 放行条件

页面可以汇报“通过”前，必须同时满足：

- 受影响页面的关键状态都已截图
- 关键截图都已用 `read` 审过
- 用户关键动作后的首屏反馈正确
- 没有阻断级布局、可读性、层级、响应式问题
- 没有被堆叠小问题压垮的主任务路径
- 证据已归档到 `.sisyphus/evidence/`

## 反模式

- `default` 通过就宣布页面通过
- 只凭 visual-loop 的结构化 `pass` 放行
- 只看 desktop，不看 mobile
- 只看整页，不看已经暴露问题的局部
