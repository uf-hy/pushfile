# 前端变更驱动视觉验收

## 目的

把前端验收从“固定脚本回放”改成“像人类一样的实时操作 + 审图回环”。

## 核心原则

1. **先看改动，再决定看什么**
2. **先走真实用户路径，再截图**
3. **先用 `read` 审关键图，再决定是否补局部图**
4. **发现问题后立刻改，再回页面继续看**
5. **证据只证明发生过，不自动等于通过**

## 用户视角 Gate（必须先回答）

在开始任何前端验收前，必须先明确以下问题：

1. **主用户是谁？**
2. **主任务是什么？**
3. **用户完成关键动作后，第一屏最应该看到什么？**
4. **当前第一屏实际给了什么？**
5. **哪些轻中度问题正在叠加，导致主任务变难？**

如果这 5 个问题答不出来，就还没有真正站到用户视角。

## 三层审查顺序（必须按顺序做）

### 第一层：任务层

先看页面是否在帮助用户完成主任务，而不是先看组件有没有坏。

必须回答：

- 这个页面是不是在服务主任务？
- 用户完成关键动作后，系统有没有立即给正确反馈？
- 主结果有没有出现在最该被看到的位置？

### 第二层：信息架构层

再看主次顺序是否对。

必须回答：

- 主结果是不是被控件抢了优先级？
- 预览 / 列表 / 当前结果是不是被放在过低位置？
- 空态、当前态、交互态之间的关系是不是顺的？

### 第三层：局部实现层

最后才看细节问题：

- 逐字换行
- 竖排
- 截断
- 重叠
- 遮挡
- 对齐
- 按钮压缩

## 适用范围

- landing / public 页面
- `/manage/*` 页面
- 任意与布局、交互、视觉层级、可读性、移动端适配有关的改动

## 执行步骤

### 第 1 步：根据改动识别影响面

先看本次改了哪些文件，再决定要去哪几个页面。

示例：

- 改 `frontend/admin/grid.*` → 必查 `/manage/grid`
- 改 `frontend/admin/manager/*` → 必查 `/manage/album`
- 改 `frontend/admin/dashboard.*` → 必查 `/manage/dashboard`
- 改 `frontend/landing/*` → 必查首页 desktop / mobile 首屏

### 第 2 步：定义本次真实用户路径

不要先写死状态名，而是先问：

- 用户会先看到什么？
- 用户会点哪里？
- 用户触发数据后页面会变成什么？
- 哪些中间态最容易藏 bug？
- 用户做完关键动作后，第一屏最该看到什么？

示例（九宫格）：

1. 打开默认页
2. 上传图片
3. 查看生成后的工作台
4. 调整参数
5. 再看预览和导出区

示例（相册后台）：

1. 打开当前路径
2. 浏览列表/网格
3. 执行搜索
4. 选择文件或文件夹
5. 切换下载模式（如果本次改到了）

### 第 3 步：用 Playwright 实时操作页面

优先使用直接 Playwright 入口，由 agent 亲自控制页面：

- 打开页面
- 真实点击
- 真实输入
- 真实上传
- 真实滚动
- 到关键状态时截图

重点：

> **不是脚本决定状态，而是当前页面与改动决定下一步。**

### 第 4 步：在关键状态截图

默认要求：

- desktop
- mobile

复杂页面至少覆盖：

- 默认态
- 当前数据态 / loaded 态
- 一次关键交互后的状态

必要时追加：

- 窄宽度 / 局部裁图
- 顶部工具条局部
- 左侧控制区局部
- 列表区域局部

### 第 5 步：主审图使用 `read`

关键截图默认全部使用 `read` 审图。

要关注：

- 主任务是否被正确服务
- 首屏反馈是否正确
- 主结果是否出现在最该被看到的位置
- 布局是否失衡
- 是否重叠、遮挡、溢出
- 中文是否难读、逐字换行、过度截断
- 视觉层级是否混乱
- 交互入口是否重复、主次不清
- 空态 / 数据态关系是否合理

### 第 5.5 步：识别“堆叠小问题”

如果 3 个以上轻中度问题都压在同一条主任务路径上，应按一个**系统级失败**处理，而不是把它们当成彼此独立的小瑕疵。

典型信号：

- 主结果不在首屏
- 控件过高、过重、过密
- 某个按钮文案被压缩
- 某块辅助信息层级混乱
- 某个局部又有挤压或截断

这些问题单独看都不致命，但叠加后会让主任务体验整体失真。

### 第 6 步：`look_at` 仅作辅助

当需要快速第二意见时，可以对同一张图再跑一次 `look_at`。

但要牢记：

- `look_at` 不是主判断
- `look_at` 不能单独充当放行依据
- 最终是否通过，仍以 `read` 的人工式审图结论为准

### 第 7 步：发现问题后继续补图

如果某个问题已经在整页图中露头，不要止步于“知道有问题”。

继续做：

- 补局部截图
- 再用 `read` 精看
- 必要时再返回页面补交互

### 第 8 步：修改后重新回环

完整链路应是：

1. 进入页面
2. 走用户路径
3. 截图
4. `read` 审图
5. 发现问题
6. 修改代码
7. 再次进入页面继续看

### 第 9 步：归档最终证据

最终只归档：

- 问题图
- 修复后的通过图
- 必要的测试输出

归档路径建议：

- `.sisyphus/evidence/<page>-<state>.png`
- `.sisyphus/evidence/<page>-<state>-fixed.png`

## 明确禁止

- 只修局部 bug，却不检查主任务是否仍然被错误的信息架构阻断
- 只看 `report.json` 就宣称页面通过
- 只审 `default` 态就代表页面整体通过
- 截图生成了，但没有明确审图记录
- 把 `.sisyphus/evidence/` 当作长期知识库
- 让 fixed-plan visual-loop 决定本次真实应该看什么
