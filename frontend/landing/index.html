02

RECENT ABILITIES // LIVE SYSTEM NOTES

页面一打开，先让人知道它不只是一个上传按钮

PushFile 的价值不在一句 slogan，而在交付之后还发生了什么： 文件能不能被组织、相册能不能被回看、短链能不能直接落地、回收站能不能兜住误删、访问量能不能被看见。

LIVE FEED 02-A

交付不是终点，后续整理、访问与恢复也在同一套面板里。

每个模块都用短句说清一件事，而不是堆形容词。

DELIVERY

相册页、短链页和文件直出分层明确

公开访问既可以走 `/album/{token}`，也可以通过 `/d/{token}` 短链进入；单图、变体和整包下载各有明确路径，不需要临时拼接方案。

  • 公开相册入口
  • 短链访问
  • 单图与 ZIP 下载
INGEST

ZIP 导入与文件夹导入直接落到目录结构

素材不是一张张手工归位。ZIP 导入和文件夹导入都能按目标目录与子目录规则落盘，隐藏文件、非法路径和超限内容会被拦住。

RECOVERY

删除先进回收站，而不是直接消失

文件与文件夹删除都优先转入回收站，后续可以恢复或彻底删除，减少误操作的不可逆风险。

OBSERVE

统计面板能看到最近活动与访问量

Dashboard 会汇总相册数量、图片数量、总访问量、今日访问量以及最近活动，让公开交付不再是“发出去之后就不知道发生了什么”。

03

WORKFLOW DOSSIERS // FOUR REAL ROUTES

它该服务的不是抽象“用户”，而是清楚的交付场景

下面四条路径对应的是这套系统真正擅长处理的事：临时交付、团队整理、内容生产和后续回看。

WORKFLOW 01 CLIENT HANDOFF

一套内容，既能公开浏览，也能一键整包拿走

上传后生成相册页，外部接收方可以直接浏览图片；需要离线交付时，再走单图下载或整包 ZIP，不需要额外导出另一套资源。

  1. 上传图片并形成相册
  2. 公开页访问走 `/album/{token}` 或短链 `/d/{token}`
  3. 接收方按单图或整包方式下载
WORKFLOW 02 TEAM ORGANIZATION

目录不是上传之后才想起整理，而是整个过程中一直可管理

文件夹可以创建、重命名、移动和删除进回收站；批量移动与批量重命名配合目录树，让素材从一开始就保持清晰，而不是靠人工补救。

  1. 按项目或客户建立目录树
  2. 通过批量移动、批量重命名保持顺序
  3. 误删时从回收站恢复
WORKFLOW 03 CONTENT PRODUCTION

导入、处理、保存放在同一条内容生产链上

ZIP 导入和文件夹导入能直接接住外部素材；九宫格模块可以预览、拆分、保存，把内容二次加工也纳入站内流程。

  1. 把素材包通过 ZIP 或文件夹整体导入
  2. 在九宫格模块里预览、拆分或保存结果
  3. 按目标目录落盘并继续交付
WORKFLOW 04 AUDIT AND RECOVERY

发出去之后还能追踪、还能恢复，才算完整工作流

访问量、最近活动、今日访问和回收站共同组成后续层。它让“交付完成”之后的阶段仍然可见，而不是只能靠猜。

  1. 在 Dashboard 查看总访问量与最近活动
  2. 识别哪些内容被访问，哪些需要回收整理
  3. 从回收站恢复误删项或彻底清理

SYSTEM MODEL // OPERATIONS, DELIVERY, PROCESSING, OBSERVABILITY

这不是一页海报，而是一套真正有模型的文件系统入口

PushFile 的核心不是单次上传，而是把进入、整理、公开交付、下载、恢复和回看串成一个连续模型。

SYSTEM NOTE 04-A

公开访问、团队整理、内容处理和统计回看是同一套系统，不是临时拼在一起的页面。

PUBLIC SURFACE

公开交付

  • 相册页与短链入口
  • 文件直出与缓存响应
  • 单图下载与整包 ZIP 下载
OPERATIONS

整理与控制

  • 目录树、子目录列表与路径安全校验
  • 批量移动、批量重命名、顺序维护
  • Token 创建、归档或彻底删除
PROCESSING

图像处理

  • 缩略 AVIF 与下载 JPEG 变体生成
  • 九宫格预览、拆分与保存
  • ZIP / 文件夹整体导入
OBSERVABILITY

统计与恢复

  • 访问量、今日访问、最近活动
  • 文件与文件夹回收站
  • 恢复与彻底删除分层处理
